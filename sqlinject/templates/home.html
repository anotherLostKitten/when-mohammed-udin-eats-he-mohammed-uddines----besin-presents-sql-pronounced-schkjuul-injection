What is a SQL Injection?

A SQL Injection is a type of attack that uses malicious SQL queries to manipulate databases and access information not meant to be displayed.

Before we go more into depth on SQL injections, lets take a little test. The image below contains four different ways to execute SQL queries. Which of the following methods is safest?


The correct answer is choice 3

Choices 1 and 2 are unsafe because the user input is inserted directly into the SQL query, allowing the user to mess with the input and create a perfectly valid, yet malicious, SQL query to be executed. Choice 4 should not be done because it misses the point of using a database. Choice 3 uses a parameterized query that sanitizes the input, preventing the user from messing with the database.

In the above image, the user input is directly inserted into the SQL query, and the query executed the query will be executed along with the user's malicious insertions.

While the above examples don't show the attacker targetting anything in particular, if the attacker included more details in their input, they could access more specific information in the database. The attacker can intentionally cause errors to find out more information about the database. The image below shows how an attacker could specifically access Mohammed Uddin's data.

SQL injections are very common and have been pulled off successfully many times on things ranging from spam bots to government sites.

The image above contains example SQL queries. The malicious user inputs that are part of the SQL injection have been highlighted


Parameterizing user inputs protects against SQL injections because the input is seperate from the query string and the input is parsed by the SQL driver. The driver sanitizes the input, preventing the input from editing the executed SQL query itself. This is safer than constructing the SQL query string and inserting the input.

Preventing SQL injections is not very difficult, and there are various methods that can be used to prevent them:

  • The user input could be restricted to alphanumeric characters and parsed through, preventing the use of quotations to construct another query to be executed.
  • Error messages could be hidden in try/except blocks, preventing the attacker from finding out information about the database.
  • The input could be parameterized, preventing an attacker from editing the SQL query string itself.
  • Dynamic SQL could be avoided when possible, which would prevent an attacker from SQL injection because there would be no input.
